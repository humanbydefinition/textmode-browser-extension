# Governance

textmode overlay uses a lightweight maintainer-led governance model. This keeps the project small enough to
move quickly while making decisions visible to contributors.

## Roles

### Users

Users install the extension, make art, test pages, report issues, request features, and give feedback. User
feedback is welcome even when it does not include code.

### Contributors

Contributors improve the project through code, docs, tests, browser QA, accessibility feedback, issue triage,
design discussion, and community support.

### Maintainers

Maintainers are responsible for project direction, review, releases, repository settings, moderation, and Code
of Conduct enforcement.

At launch, the primary maintainer is `@humanbydefinition`.

## Decision-Making

Maintainers make final decisions, but decisions should be public when they affect users or contributors.

- Small fixes may be decided in pull request review.
- Features, browser-support changes, permission changes, and architecture changes should start in an issue.
- Larger design decisions should be summarized before implementation.
- Accepted decisions should be reflected in docs, tests, ADRs, or release notes when appropriate.

Consensus is preferred, but unanimity is not required. Maintainers may decline proposals that do not fit the
project vision, maintenance capacity, browser target, dependency policy, privacy posture, or extension-store
requirements.

## Becoming More Trusted

Contributors can earn more trust by repeatedly making helpful, focused contributions.

Useful signals include:

- Clear bug reproductions.
- Small pull requests that pass checks.
- Thoughtful reviews.
- Helpful issue triage.
- Browser QA notes.
- Improvements to docs and tests.
- Constructive public design discussion.

As the contributor community grows, maintainers may invite trusted contributors to help with triage, review, or
specific ownership areas.

## Review and Merge

- External pull requests target `main` unless maintainers announce a dedicated development branch.
- Maintainers review for correctness, scope, tests, docs, privacy, browser support, and fit with project
  direction.
- Maintainers may squash external pull requests with a clear commit title.
- Release tags and extension packages are created by maintainers after checks pass.

## Code of Conduct

All project spaces follow [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md).

Maintainers are responsible for enforcing the Code of Conduct. Reports may be sent privately using the contact
information in the Code of Conduct. Enforcement actions may include warnings, comment moderation, temporary
restrictions, or removal from project spaces.

## Changes to Governance

Governance changes should be proposed publicly through an issue or pull request. Maintainers make the final
decision after considering contributor feedback.
